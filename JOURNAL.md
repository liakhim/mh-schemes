## Актуализация: автоподбор модулей и PRO/ECOsmart

- Модули, которые `/selection` добавляет для покрытия дефицита портов, помечаются `_auto_source: 'selection-required-module'`.
- К автоматическим относятся `BL2`, `RL6`, `RL6S`, `IO4`, `DI6`, `RL2`, `RL2S`, `NTC-1-wire` и `RDT2`, если они добавлены подборщиком.
- После изменения состава оборудования подбор повторно рассчитывает потребность и удаляет только лишние модули с этой меткой. Подходящие автоматические модули сохраняют прежние `id`.
- Модули без этой метки считаются добавленными вручную и не удаляются при уменьшении нагрузки. Это также относится к схемам, сохранённым до появления метки.
- При переключении `PRO -> ECOsmart` EXT-модули разворачиваются обратно в публичные списки устройств, несовместимые модули удаляются, термостаты с датчиком пола переносятся в `controller.ext_devices`, а для второго BUS-котла добавляется внутренний `controller.ecosmart_bl2`.
- При переключении `ECOsmart -> PRO` `controller.ext_devices` возвращаются в `wired_devices` с `connection_type='1-wire'`, внутренний `controller.ecosmart_bl2` удаляется, а датчик тупого котла восстанавливается в `sensors`, если он нужен.
- Если состав совместим с обоими контроллерами, переключатель `PRO` / `ECOsmart` остаётся доступен в обоих состояниях.
- Preview ECOsmart в JSON проходит тот же путь переключения и пересчёта модулей, что и реальный выбор контроллера.
- Единственное актуальное хранилище overlay-модуля: `controller.ecosmart_bl2`. Корневое `ecosmart_bl2` читается только как legacy-формат и при загрузке переносится внутрь controller.
- Отложенная нормализация `/selection` применяет результат к актуальному состоянию схемы через функциональный state update; она не может перезаписать только что добавленное пользователем устройство устаревшим состоянием после переключения controller.
- Очистка датчика тупого котла для ECOsmart идемпотентна: если такого датчика нет, она возвращает исходный объект схемы и не запускает лишний цикл нормализации.

## Актуализация: запорный клапан на RL2S

- `valve` с `connection_type='double_relay'` поддерживается не только ручным добавлением, но и автоматической балансировкой в `rl2s.relay_s_devices`.
- Клапан занимает оба RELAY-S слота `RL2S`; renderer показывает общий `L`, две линии от `RELAY-IN-1` и `RELAY-IN-2` к `RELAY-S-1-B` и `RELAY-S-2-B`. Изображение целиком и пропорционально вписывается в обычный relay-слот.

## Актуализация: первый 1-wire на PRO

- Минимальный уровень горизонтальных участков трёх линий controller -> первое устройство 1-wire PRO рассчитывается от нижней границы реально отображаемого controller-слота, а не по фиксированному набору выносов.
- Учитываются `MODBUS`, `DI`, пустой `AI`, `4-20`, батарея, `RELAY`, `BUS` и горизонтальные участки EXT-соединений; пустые слоты попадают в расчёт только при включённом `Show empty slots`.
- Минимальный зазор между нижней границей найденного слота и горизонтальным участком `V+`: `1 * indent`; `DAT` проходит ещё на `1 * indent` ниже, `GND` - на `2 * indent` ниже.

## Актуализация: тупой котёл и RELAY-S

- ВАЖНО: тупой котёл (`stupid`) не может садиться на RELAY-S порты — ни при подборе/балансировке, ни при отрисовке, ни при добавлении через слоты.
- Пункт «Тупой котёл» скрыт в меню relay-s слотов: controller relay-s (`pro`), `rl6s.relay_s_devices`, `rl2s.relay_s_devices`; обработчики добавления дополнительно блокируют тип `stupid` для этих линий.
- `relayDeviceBalancer.supportsRelayS` возвращает `false` для тупого котла: балансировка никогда не размещает и не оставляет его в relay-s линиях.
- Тупой котёл, застрявший в `controller.relay_s_devices` сохранённой схемы, при материализации переносится в `controller.relay_devices` с `connection_type='relay'`.
- Тупой котёл, застрявший в `rl6s`/`rl2s` `relay_s_devices`, переносится в обычные relay-линии (controller, затем `rl6`); если мест нет — возвращается в верхнеуровневый `boilers`.
- На `/selection` relay-котлы и так считаются строго relay-устройствами (не участвуют в relay-s capacity) — без изменений.

## Актуализация: ecosmart слоты и режим монтажа

- Инфоблок слота насоса бойлера ГВС `ecosmart` отображается только при занятом слоте; над пустым слотом инфоблок не рисуется даже при `Show empty slots` (как у остальных per-role слотов).
- Hover-крестик удаления у слотов `ecosmart` (leak-sensor, discrete DI, верхние NTC-слоты) обрабатывается на уровне Konva `Group`, а не `Rect` слота: наведение на сам крестик не снимает hover и не прячет кнопку.
- Занятый controller RELAY-слот `ecosmart` (тупой котёл) показывает hover-крестик удаления; удаление вычищает устройство по id из `controller.relay_devices`, `boilers` и `wired_devices`.
- В режиме монтажа семантический тег в имени порта — часть его идентичности: `RELAY-1-A` (реле котла) и `RELAY-1-A BOILER-GVS` (насос ГВС) — разные физические пины; дедупликация портов выполняется по полному имени, а не по первому токену.
- В режиме монтажа занятость и подписи тегированных портов `ecosmart` проверяются по per-role линиям контроллера: `BOILER-GVS` -> `relay_boiler_gvs_devices`, `220PUMP` (RELAY-2/5/3) -> `relay_220pump_devices`/`relay_220pump5_devices`/`relay_220pump3_devices`, `VALVE` -> `relay_s_valve_devices`, `CASCADE` -> `strategy_sensor_devices`, `BOILER` (NTC-2) -> `boiler_sensor_devices`, `MIXING` (NTC-3/4) -> `mixing_ntc_devices`.
- Нетегированные `RELAY-6-A/B` и `RELAY-4-A/B` у `ecosmart` — слоты сервоприводов смесителей: занятость по `220_servo_devices[0]` / `220_servo_devices[1]`.
- `DI-IN-2-DI` у `ecosmart` — вход датчика протечки: занятость по `leak_sensor_devices[0]`.
- Хвост и лычка рисуются только на терминалах `A`/`B`; `GND`/`V+` — вспомогательные, без хвостов и лычек.
- `EXT-OUT` контроллера считается занятым также при наличии EXT-термостатов (`controller.ext_devices`); если следующего модуля в цепочке нет, подпись берётся из первого EXT-термостата.
- EXT-цепочка в режиме монтажа: контроллер -> EXT-модули -> EXT-термостаты. `moduleNextLabel` последнего EXT-модуля (и контроллера при отсутствии модулей) указывает на первый EXT-термостат линии — его `EXT-OUT` получает хвост и лычку с подписью термостата.
- Клеммы аккумулятора ИБП `ACID-BAT-V+` и `ACID-BAT-GND` в режиме монтажа всегда считаются занятыми и получают хвост и лычку с подписью «Батарея».
- DI-линии коммутации ИБП (`DI-OUT-1`, `DI-OUT-2`) в режиме монтажа всегда считаются занятыми и получают хвост и лычку с подписью контроллера, к DI-входам которого они подключены.
- Встречная сторона: при наличии ИБП первые два DI-входа контроллера (`DI-IN-1`/`DI-IN-2` у pro, первая пара `DI-OUT` у smart2) считаются занятыми и получают хвост и лычку «ИБП»; если DI-вход занят реальным устройством, приоритет у его подписи.
- Линия `4-20` в режиме монтажа учитывает legacy-ключ `devices420` наряду с `devices_420`.
- При смене контроллера с `ecosmart` на другой на `/selection` внутренние материализации разворачиваются: термостаты из `controller.ext_devices` возвращаются в `wired_devices` с `connection_type='1-wire'`, `controller.ecosmart_bl2` удаляется, датчик тупого котла (`flask-sensor-stupid-boiler`) восстанавливается в `sensors`, если в схеме есть тупой котёл.

## Актуализация: публичный incomingScheme и внутренние ecosmart-линии

- Специальные `ecosmart` controller-линии являются внутренней материализацией для отрисовки и не должны быть видны в публичном `incomingScheme` даже после распределения балансировщиками.
- При показе JSON и сохранении внутренние `ecosmart`-устройства сериализуются обратно в публичные списки: wired-устройства в `wired_devices`, датчики в `sensors`.
- Внутренние wired-линии `ecosmart`: `controller['220_servo_devices']`, `controller.relay_s_valve_devices`, `controller.relay_boiler_gvs_devices`, `controller.relay_220pump_devices`, `controller.relay_220pump5_devices`, `controller.relay_220pump3_devices`.
- Внутренние sensor-линии `ecosmart`: `controller.leak_sensor_devices`, `controller.strategy_sensor_devices`, `controller.boiler_sensor_devices`, `controller.mixing_ntc_devices`.
- `controller.ecosmart_bl2` также считается внутренним render-полем и не должен сохраняться как публичная линия входного контракта.
- Canvas продолжает строиться по внутреннему `scheme` после материализации; очистка применяется только к публичной сериализации JSON/save.

## Актуализация: DI-инфоблоки

- Над занятыми DI-слотами для `discrete_pool`, `discrete_fire_alarm`, `discrete_signal`, `discrete_ventilation` отображаются инфоблоки с названием устройства и порядковым номером среди устройств этого типа.
- Текст инфоблока размещённых в линиях controller/EXT/DI-module устройств материализуется в поле `title` самого устройства; legacy-поле `titile` поддерживается только как fallback.
- Текст инфоблока редактируется двойным кликом/двойным тапом по подписи и сохраняется обратно в `device.title`.
- Пустые controller DI-слоты `pro` при `Show empty slots` показывают кнопку `+`; через меню можно добавить `discrete_pool`, `discrete_fire_alarm`, `discrete_signal`, `discrete_ventilation`, `leak-sensor` в `controller.di_devices[0..1]`.
- Пустые отдельные controller DI-слоты `smart2` (не слоты `di_modules`) при `Show empty slots` показывают кнопку `+`; через меню можно добавить `discrete_pool`, `discrete_fire_alarm`, `discrete_signal`, `discrete_ventilation`, `leak-sensor` в `controller.di_devices`.
- Занятые controller DI-слоты `pro`, `smart2` и отдельный discrete DI-слот `ecosmart` на hover показывают кнопку удаления; удаление очищает конкретный индекс `controller.di_devices` без сдвига соседних устройств.
- Если `leak-sensor` занимает controller DI-слот `pro`, используется изображение `resources/assets/sensors/leakSensorLeftPort.svg`.
- Вертикальный gap между channel-слотами `io4` увеличен до `8 * indent`, чтобы инфоблоки занятых слотов не накладывались на соседние устройства.
- У занятых `io4.channel_devices` слотов на hover отображается кнопка удаления; удаление очищает конкретный channel-индекс без сдвига соседних устройств.
- Пустые `io4.channel_devices` слоты показывают номер канала `1..4` в правом верхнем углу.
- Над занятыми `io4.channel_devices` слотами для `discrete_pool`, `discrete_fire_alarm`, `discrete_signal`, `discrete_ventilation` отображается редактируемый инфоблок с названием устройства.

## Актуализация: rl2s own RELAY-S slots

- Пустые собственные RELAY-S слоты модуля `rl2s` при `Show empty slots` показывают кнопку `+`.
- Через меню в `rl2s.relay_s_devices` можно добавить `zoneServo` с `connection_type='relay-s'`, `220servo` с `connection_type='double_relay'`, `valve` с `connection_type='double_relay'`.
- `220servo` и `valve` занимают два соседних RELAY-S слота `rl2s`; добавление доступно только если выбранный слот и следующий слот свободны.
- Меню собственных RELAY-S слотов `rl2s` отделено от меню добавления DI-модулей `rl2`/`rl2s`.
- Инфоблоки собственных RELAY-S слотов `rl2s` компактные: ширина равна ширине слота, высота 12px, позиция над слотом как у relay-слотов EXT-модулей.

## Актуализация: selection relay|relay-s capacity

- На `/selection` `boilerPump`/`boiler-pump` с `connection_type='relay|relay-s'` считается гибким устройством: сначала занимает доступные relay-слоты, остаток учитывается как relay-s.
- Новый целевой формат для `220pump`/`pump-220v`: `connection_type='relay|relay-s'`; насос 220V может занимать обычные RELAY и RELAY-S слоты.
- Автоподбор EXT-модулей для `pro`/`ecosmart` пересчитывает relay-s дефицит после добавления `rl6`, потому что дополнительная relay-capacity может забрать часть гибких `relay|relay-s` устройств и убрать лишнюю потребность в `rl6s`/дополнительном модуле.
- Если controller RELAY-слот занят `boilerPump`/`boiler-pump`, линия коммутации выходит из порта насоса сначала вправо на `1 * indent`, затем вниз до горизонтального участка, затем вправо/в сторону вертикали соответствующего controller RELAY-порта и поднимается вверх в порт.
- Если controller RELAY-S слот `pro` занят `boilerPump`/`boiler-pump`, линия коммутации выходит из изображения насоса сразу в сторону до вертикали соответствующего controller RELAY-S порта и затем спускается/поднимается в порт.
- Вертикальный зазор между слотами controller RELAY-S линии `pro`: `4 * indent`.
- Занятые controller RELAY-S слоты `pro` на hover показывают кнопку удаления; удаление убирает устройство из `controller.relay_s_devices`.

## Актуализация: ecosmart NTC-линии

- `flask-sensor-strategy` и `flask-sensor-gvs-boiler` должны приходить с `connection_type='1-wire|ntc'` как целевым глобальным форматом.
- Для `ecosmart` такие датчики подключаются как `ntc`; для остальных контроллеров остаются обычными `1-wire`.
- Для `ecosmart` `flask-sensor-strategy` размещается в `controller.strategy_sensor_devices[0]`.
- Для `ecosmart` `flask-sensor-gvs-boiler` размещается в `controller.boiler_sensor_devices[0]`.
- Для `ecosmart` два `mixing-ntc-sensor` берутся из `220servo.additions` и размещаются в `controller.mixing_ntc_devices[0..1]`.
- Перенесённые датчики удаляются из исходных `sensors`/`additions`.
- Над `ecosmart` отрисовываются 4 вертикальных слота: снизу `strategy-sensor-line`, выше `boiler-sensor-line`, выше два слота `mixing-ntc-sensor`.
- Размер каждого верхнего слота `9 * indent` × `2 * indent`; нижняя грань нижнего слота находится на `4 * indent` ниже верхней грани контроллера; gap между слотами `3 * indent`; правые грани слотов совпадают с правой гранью контроллера.
- Пустые верхние NTC-слоты `ecosmart` отображаются всегда, независимо от `Show empty slots`.
- Верхние NTC-слоты `ecosmart` коммутируются к реальным NTC-портам controller: strategy -> `NTC-1-A/B CASCADE`, boiler -> `NTC-2-A/B BOILER`, mixing 1 -> `NTC-3-A/B MIXING`, mixing 2 -> `NTC-4-A/B MIXING`.
- Для всех занятых верхних NTC-слотов `ecosmart` всегда используется `resources/assets/sensors/ntcSensorLeftPort.svg`, независимо от типа датчика.

## Актуализация: ecosmart BUS-слоты

- BUS-слоты `ecosmart` остаются смещёнными левее на `4.5 * indent` и дополнительно подняты вверх на `6 * indent`.

## Актуализация: leak-sensor

- Новый датчик протечки: `{ id: 13, device_type: 'sensor', type: 'leak-sensor', connection_type: 'di' }`.
- `leak-sensor` участвует в DI-балансировке для всех контроллеров: controller DI, `io4.channel_devices`, `di6.channel_devices`.
- Для `ecosmart` первый `leak-sensor` занимает собственную `leak_sensor_line` и хранится в `controller.leak_sensor_devices[0]`.
- `ecosmart leak_sensor_line` подключается к портам `DI-IN-2-DI`, `DI-IN-2-V+`, `DI-IN-2-GND`.
- Слот `ecosmart leak_sensor_line` расположен над контроллером: нижняя грань на `12 * indent` выше верхней грани контроллера, размер `7 * indent` на `7 * indent`.
- Над занятым слотом отображается инфоблок `Датчик протечки N`, где `N` - порядковый номер датчика протечки в системе.
- Линии от портов датчика протечки сначала выходят горизонтально в сторону до вертикали соответствующего порта controller, затем спускаются в порт controller.
- Если порты controller правее слота, используется `resources/assets/sensors/leakSensorRightPort.svg`; если левее слота, используется `resources/assets/sensors/leakSensorLeftPort.svg`.
- Под `ecosmart leak_sensor_line` расположен ещё один собственный DI-слот controller для `discrete_pool`, `discrete_fire_alarm`, `discrete_signal`, `discrete_ventilation`.
- Этот `ecosmart` DI-слот имеет размер `7 * indent` на `2 * indent`, смещён на `1 * indent` левее базовой позиции, использует изображения с правыми портами и коммутируется одним проводом в controller port `DI-IN-1`.
- Инфоблок над этим `ecosmart` DI-слотом не отображается.

## Актуализация: ecosmart valve relay-s line

- Для `ecosmart` `220servo` с `connection_type='double_relay'` размещается только в отдельной controller-линии `controller['220_servo_devices']`.
- Слот `ecosmart 220_servo_devices` расположен в левой нижней части controller со смещением на `6 * indent` правее базовой левой позиции и подъёмом на `6 * indent` выше прежней позиции; используется `resources/assets/servo/220servoRightPorts.svg`.

- Линия использует изображение `resources/assets/modules/valve/valveLeftPort.svg`.
- Коммутация выполняется в controller-порты `RELAY-S-1-V+ VALVE`, `RELAY-S-1-A VALVE`, `RELAY-S-1-B VALVE`, `RELAY-S-1-GND VALVE`.
- Слот расположен справа от controller: левая грань слота начинается от правой грани controller, нижняя грань слота на `6 * indent` выше нижней грани controller.

## Актуализация: ecosmart boiler GVS pump relay line

- Для `ecosmart` `boilerPump`/`boiler-pump` с `connection_type='relay|relay-s'` размещается в отдельной линии `controller.relay_boiler_gvs_devices`.
- Линия расположена под слотом `relay_s_valve_devices` с дополнительным опусканием `2 * indent`, ширина слота `8 * indent`, используется `resources/assets/pumps/boilerPumpLeftPort.svg`.
- Коммутация насоса бойлера ГВС: `RELAY-1-A BOILER-GVS` -> `RELAY-IN A`, `RELAY-1-B BOILER-GVS` -> `RELAY-IN B`, `RELAY-1-GND BOILER-GVS` -> `RELAY-IN-GND`.
- Линии от портов изображения насоса бойлера сначала выходят сразу влево, затем идут к соответствующим controller-портам.

## Актуализация: ecosmart 220V pump relay line

- Для `ecosmart` `220pump`/`pump-220v` с `connection_type='relay'` размещается в отдельной линии `controller.relay_220pump_devices`.
- Линия расположена под слотом `relay_boiler_gvs_devices` и использует `resources/assets/pumps/220pumpLeftPort.svg`.
- Коммутация насоса 220V: `RELAY-2-A 220PUMP` -> `RELAY-IN A`, `RELAY-2-B 220PUMP` -> `RELAY-IN B`, `RELAY-2-GND 220PUMP` -> `RELAY-IN-GND`.
- Если `ecosmart relay_220pump_devices` уже занят, следующий `220pump`/`pump-220v` с `connection_type='relay'` размещается в `controller.relay_220pump5_devices` под слотом `220_servo_devices`; слот поднят на `6 * indent` выше прежней позиции.
- `ecosmart relay_220pump5_devices` использует `resources/assets/pumps/220pumpRightPort.svg`; коммутация: `RELAY-IN A` -> `RELAY-5-A 220PUMP`, `RELAY-IN B` -> `RELAY-5-B 220PUMP`, `RELAY-IN-GND` -> `RELAY-5-GND 220PUMP`.
- Если `ecosmart relay_220pump5_devices` уже занят, следующий `220pump`/`pump-220v` с `connection_type='relay'` размещается в `controller.relay_220pump3_devices` под вторым слотом `220_servo_devices` и дополнительно опущен на `2 * indent`.
- `ecosmart relay_220pump3_devices` использует `resources/assets/pumps/220pumpRightPort.svg`; коммутация: `RELAY-IN A` -> `RELAY-3-A 220PUMP`, `RELAY-IN B` -> `RELAY-3-B 220PUMP`, `RELAY-IN-GND` -> `RELAY-3-GND 220PUMP`.
- В текущем `ecosmart.svg` физических классов `RELAY-3-*` нет, поэтому renderer использует виртуальные позиции нижней клеммной группы.

## Актуализация: ecosmart first 1-wire link

- Для `ecosmart` у линии соединения controller и первого 1-wire устройства вертикальный участок смещён левее на `6 * indent` от портов controller.
- Для `ecosmart` первое 1-wire устройство и первый пустой слот при пустой линии расположены на `5 * indent` ниже базовой позиции.

## Глобальные переменные

- `indent` = `'8px'` — расстояние между линиями и величина для выноса линий под или в сторону от устройств
- `module_height` = `'200px'` — высота любого модуля в системе
- `din` = `'40px'` — часть ширины модуля или контроллера

## Запорный клапан

- Запорный клапан хранится в `wired_devices` как объект:
```js
{
    id: 0,
    device_type: 'equipment',
    type: 'valve',
    connection_type: 'double_relay',
    additions: []
}
```
- Занимает два соседних порта `relay` / `relay-s`
- Если соединяемые порты левее слота, используется `resources/assets/modules/valve/valveLeftPort.svg`
- Если соединяемые порты правее слота, используется `resources/assets/modules/valve/valveRightPort.svg`
- Коммутация `valve` выполняется через порты SVG `RELAY-IN-1` и `RELAY-IN-2`

## Сервопривод зоны

- Сервопривод зоны хранится в `wired_devices` как объект:
```js
{
    id: 0,
    device_type: 'equipment',
    type: 'zoneServo',
    connection_type: 'relay',
    additions: []
}
```
- Коммутируется одной линией с relay-портом контроллера или модулей `rl6`, `rl6s`, `rl2`, `rl2s`
- Занимает один relay-слот
- Участвует в распределении relay-балансировщиком из `wired_devices`
- Если слот расположен правее портов, с которыми коммутируется, используется `resources/assets/servo/zoneServoLeftPort.svg`
- Если слот расположен левее портов, с которыми коммутируется, используется `resources/assets/servo/zoneServoRightPort.svg`

## Понятие "линия"

**Линия** — зона расположения разных по типу слотов, набор которых зависит от вида контроллера.

- Каждый контроллер имеет свой набор линий
- Внутри линии располагаются слоты соответствующих типов
- Тип линии определяет, какие модули/устройства могут быть подключены
- Устройство, требующее несколько слотов, не размещается в линии, если свободных соседних слотов меньше требуемого количества; например `220servo`/`valve` с `double_relay` требуют 2 relay-слота и не могут быть размещены на `go` с одним relay-каналом

### Понятие "инфоблок"

**Инфоблок** — прямоугольный блок со скруглёнными углами, содержащий подпись.

- Размещается над соответствующими линиями/элементами для пояснения их назначения
- Шрифт подписи: 4px
- Скругление углов: 2px
- Нижний margin: 8px
- Padding: 6px по бокам
- Фон: голубой (`#dce8f5`), обводка: светло-синяя (`#8ab4d6`)

### Инфоблоки в системе

| Расположение | Текст |
|---|---|
| Над беспроводной линией | "Беспроводные устройства" |

## Беспроводная линия (wireless_line)

- Расстояние считается между нижней гранью беспроводной линии и верхней гранью контроллера
- Расстояние зависит от типа контроллера:
  - `go`, `go+`: `0.6 * module_height`
  - `pro`: `1.85 * module_height`
  - остальные контроллеры: `1 * module_height`
- Визуально нижняя грань беспроводных слотов лежит на уровне `-wireless_gap`, где `wireless_gap` определяется типом контроллера
- Нижний край инфоблока `Беспроводные устройства` расположен на `40px` выше верхней грани беспроводной линии
- Если беспроводная линия пустая и выключены `Show empty slots` и `Show line frames`, инфоблок `Беспроводные устройства` не отображается
- Если беспроводная линия пустая, но включён `Show empty slots` или `Show line frames`, инфоблок `Беспроводные устройства` отображается
- Пустая беспроводная линия отображается при включённом `Show empty slots` или `Show line frames`
- Содержит беспроводные слоты (wireless slot), заполняется из `incomingScheme.wireless_devices`
- Пустой слот: квадрат 80×80px со скруглёнными углами (cornerRadius: 10), внутри круглая кнопка 32px с символом "+"
- Иконка "+" центрирована внутри кнопки
- Нажатие на "+" открывает контекстное меню в позиции клика со списком устройств для добавления

### Текущее поведение беспроводной линии (актуализация)

- Источник данных: `incomingScheme.wireless_devices`
- Каждый выбранный элемент занимает отдельный слот в линии
- Слоты вычисляются последовательно по ширине (без наложений), с учётом индивидуальной ширины слота устройства
- Позиции занятых слотов можно менять перетаскиванием (draggable)
- Кнопка `Reset positions` сбрасывает пользовательские смещения слотов

### Отображение занятых слотов

- По умолчанию у занятого слота скрыты фон, обводка и кнопка удаления
- На hover у занятого слота показываются:
  - фон и обводка слота
  - кнопка удаления (круг с крестиком в правом верхнем углу)

### Чекбоксы интерфейса

- `Show ports` — отображает порты контроллера и устройств (красные точки)
- `Show empty slots` — отображает только пустые слоты:
  - пустой слот добавления в беспроводной линии
  - пустой one-wire слот справа от термостата
  - пустой MODBUS-слот контроллера `pro`
  - для `smart2`: если все `DI` порты контроллера заняты, пустые слоты `power` и `DI` линий не отображаются

|---|---|
| `outdoor-temperature-sensor` | `resources/assets/sensors/wirelessOutdoorSensor.svg` |
| `wall-temperature-sensor` | `resources/assets/sensors/wirelessWallSensor.svg` |
| `thermostat` | зависит от `color` (см. ниже) |

### Беспроводной термостат

- Термостат поддерживает цвета `black`, `white`, `gray`
- Цвет берётся из поля `color` во входящем объекте устройства
- Соответствие SVG:
  - `black` -> `resources/assets/thermostats/black/thermostat_black.svg`
  - `white` -> `resources/assets/thermostats/white/thermostat_white.svg`
  - `gray` -> `resources/assets/thermostats/gray/thermostat_gray.svg`
- Если цвет отсутствует или неизвестен, используется `black`

### One-wire слот термостата (датчик пола)

- У термостата есть встроенный one-wire слот справа от изображения
- Размер one-wire слота: 70×70px
- One-wire слот может содержать только одно устройство
- Контекстное меню one-wire слота содержит один пункт: `Датчик пола`
- После выбора в `additions` термостата сохраняется один элемент типа `floor-sensor`
- SVG датчика пола: `resources/assets/sensors/flaskSensor.svg`
- Если слот занят датчиком пола, фон и обводка one-wire слота не отображаются (видно только изображение)
- Беспроводной термостат берёт датчик пола из собственного `additions`
- Проводной термостат с датчиком пола в `additions` при балансировке 1-wire разворачивается в группу: сам термостат + датчик пола в той же 1-wire линии; размещённый датчик удаляется из `additions` материализованного термостата, чтобы не было дублирования

### Коммутация термостат <-> датчик пола

- Правило направления линии: из порта линия выходит строго влево, вправо или вниз
- Вынос линии считается от грани устройства, а не от точки порта
- Для пары `термостат <-> датчик пола` используются порты:
  - `1-WIRE-GND` с выносом вниз `3 * indent`
  - `1-WIRE-DAT` с выносом вниз `2 * indent`
  - `1-WIRE-V+` с выносом вниз `1 * indent`
- Для каждой линии маршрут: от порта термостата вниз до уровня выноса -> горизонтально к X порта датчика -> вертикально к порту датчика
- При включённом `Show ports` отображаются порты и термостата, и датчика пола

## 1-wire линия термостатов (этап 1)

- Линия `1-wire` устройства строится под устройством, опираясь на расположение портов `1-WIRE-*`
- Точка старта линии смещается вправо на `1 * indent`
- Базовый старт по `Y` берётся от нижнего уровня группы `1-WIRE` портов
- Для контроллеров `go` и `go+` первый слот (или первый занятый слот) имеет дополнительный отступ вниз `0.5 * module_height` от уровня `1-WIRE` портов
- При включённом `Show empty slots` отображается первый пустой слот `1-wire` линии

### 1-wire линия (актуальные правила)

- Ограничение линии контроллера: максимум `6` устройств
- Источники устройств для линии:
  - `wired_devices` с `type = thermostat` (проводные термостаты)
  - `one_wire_modules` (`ntc-1-wire`, `rdt/rdt2`)
  - Поддерживается alias: `rdt` -> `rdt2`
- Если первый слот занят, отображается следующий пустой слот
- Левый край следующего слота: на `2 * indent` правее правого края предыдущего устройства
- Если в цепочке присутствует `ntc-1-wire`, для такого устройства применяется дополнительный отступ сверху `12 * indent`
- Если в цепочке присутствует `ntc-1-wire`, для такого устройства применяется дополнительный боковой отступ:
  - слева `10 * indent`
  - справа `10 * indent`
- Вертикальный шаг:
  - по умолчанию: ниже нижней линии предыдущего устройства + `0.25 * module_height`
  - если предыдущее устройство — `ntc-1-wire` или `rdt2`: ниже нижнего края модуля + `0.25 * module_height`

### Порты и коммутация 1-wire линии

- Пустой слот имеет фейковые порты в левой нижней части: `1-WIRE-V+`, `1-WIRE-DAT`, `1-WIRE-GND`
- Если слот занят устройством, используются порты из SVG устройства (фейковые порты не актуальны)
- Цвета линий:
  - `1-WIRE-V+` — красный
  - `1-WIRE-DAT` — жёлтый
  - `1-WIRE-GND` — чёрный
- Вынос линий для подключения к слоту считается от целевого слота:
  - `V+`: `1 * indent` ниже слота
  - `DAT`: `2 * indent` ниже слота
  - `GND`: `3 * indent` ниже слота
- Если целевой слот занят термостатом, соединение выполняется по правилу термостата: линия приходит на уровень порта термостата (без выноса ниже слота)

### Минимальные выносы под контроллером для 1-wire (`pro`)

- Правило действует для соединений контроллера `pro` с первым устройством в линии `1-wire`
- Минимальный уровень горизонтального участка линии считается от нижней границы отрисованного контроллера:
  - `1-WIRE-GND`: `5 * indent`
  - `1-WIRE-DAT`: `4 * indent`
  - `1-WIRE-V+`: `3 * indent`
- Если у контроллера занят слот `RELAY` линии, для первого устройства `1-wire` используется увеличенный минимальный уровень:
  - `1-WIRE-GND`: `7 * indent`
  - `1-WIRE-DAT`: `6 * indent`
  - `1-WIRE-V+`: `5 * indent`
- Если вынос, рассчитанный по общим правилам, меньше этих значений, используется минимум от контроллера

### Минимальные выносы под контроллером для 1-wire (`smart2`)

- Правило действует для соединений контроллера `smart2` с первым устройством в линии `1-wire`
- Минимальный уровень горизонтального участка линии под нижней границей контроллера зависит от количества устройств в `DI` линии:
  - Если в `DI` линии 2 устройства:
    - `1-WIRE-GND`: `11 * indent`
    - `1-WIRE-DAT`: `10 * indent`
    - `1-WIRE-V+`: `9 * indent`
  - Если в `DI` линии 1 устройство:
    - `1-WIRE-GND`: `9 * indent`
    - `1-WIRE-DAT`: `8 * indent`
    - `1-WIRE-V+`: `7 * indent`
  - Если в `DI` линии нет устройств:
    - `1-WIRE-GND`: `5 * indent`
    - `1-WIRE-DAT`: `4 * indent`
    - `1-WIRE-V+`: `3 * indent`
- Если вынос, рассчитанный по общим правилам, меньше этих значений, используется минимум от контроллера

### Минимальные выносы под контроллером для 1-wire (`go`, `go+`)

- Правило действует для соединений контроллеров `go` и `go+` с первым устройством в линии `1-wire`
- Минимальный уровень горизонтального участка линии под нижней границей контроллера:
  - `1-WIRE-GND`: `7 * indent`
  - `1-WIRE-DAT`: `6 * indent`
  - `1-WIRE-V+`: `5 * indent`
- Если вынос, рассчитанный по общим правилам, меньше этих значений, используется минимум от контроллера

### NTC-1-wire и RDT2

- Оба модуля имеют порты `1-WIRE-*` с классами направления `IN` и `OUT`
- Размер слота для `ntc-1-wire` и `rdt2` принимается по реальному размеру изображения (без растяжения)
- Для соединения со следующим устройством используются `OUT` порты модуля
- Для подключения модуля как целевого устройства используются `IN` порты
- Для точной привязки линий и точек портов используются якорные координаты портов (`IN/OUT`) для `ntc-1-wire` и `rdt2`
- Над занятыми слотами `ntc-1-wire` и `rdt2` инфоблок не отображается

#### Дополнительные порты NTC-1-wire

- В SVG модуля `ntc-1-wire` доступны дополнительные порты:
  - `NTC-1-A`, `NTC-1-B`
  - `NTC-2-A`, `NTC-2-B`
  - `NTC-3-A`, `NTC-3-B`
  - `NTC-4-A`, `NTC-4-B`
  - `NTC-5-A`, `NTC-5-B`
  - `NTC-6-A`, `NTC-6-B`

### NTC-линии NTC-1-wire (этап 1)

- У модуля `ntc-1-wire` есть первая `NTC`-линия
- У модуля `ntc-1-wire` есть первая и вторая `NTC`-линии
- Первая `NTC`-линия расположена слева сверху от модуля
- Вторая `NTC`-линия расположена справа сверху от модуля
- Слоты в линии расположены вертикально друг под другом
- Количество слотов первой `NTC`-линии: `3`
- Размер слота NTC-линии:
  - ширина: `11 * indent`
  - высота: `3 * indent`
- Правая граница первой NTC-линии выравнивается по левой границе модуля `ntc-1-wire`
- Нижняя граница первой NTC-линии находится на уровне верхней границы модуля `ntc-1-wire`
- Пустые слоты NTC-линии отображаются только при включенном `Show empty slots`
- Для первой `NTC`-линии используется датчик с изображением `resources/assets/sensors/ntcSensorRightPort.svg`
- Для второй `NTC`-линии используется датчик с изображением `resources/assets/sensors/ntcSensorLeftPort.svg`
- Изображение NTC-датчика отображается с сохранением пропорций (без растяжения по высоте)
- Над занятым слотом NTC-линии отображается подпись `NTC-датчик N`, где `N` — порядковый номер датчика в системе
- В порядковый номер `NTC-датчик N` входят только датчики `type = ntc-sensor`; `boiler-ntc-sensor` не учитывается и идёт отдельно
- Линии коммутации от портов модуля `NTC-*-B` рисуются цветом `#464EE3`
- Линии коммутации от портов модуля `NTC-*-A` рисуются чёрным цветом `#212121`
- Над самим модулем `ntc-1-wire` инфоблок не отображается
- Датчики из `sensors` с `type = ntc-sensor` и `connection_type = ntc` после балансировки `ntc-1-wire` модулей распределяются между всеми модулями `ntc-1-wire`
- Назначенные NTC-датчики сохраняются в собственных линиях модуля: `ntc1_devices` и `ntc2_devices`, до `3` датчиков в каждой линии
- Балансировка NTC-датчиков выбирает наименее заполненный `ntc-1-wire` модуль, чтобы распределять датчики между модулями, а не заполнять только первый модуль
- Назначенные в `ntc-1-wire` датчики удаляются из верхнего поля `sensors`, чтобы не дублироваться в других линиях

## Relay линия (`go`, `go+`, `smart2`, `pro`, этап 1)

- Для `go`, `go+`, `smart2` и `pro` добавлена базовая relay-линия со слотом добавления устройства (`+`)
- Количество слотов relay-линии равно количеству relay-портов контроллера-владельца линии
- Пустой слот relay-линии для всех контроллеров отображается только при включенном `Show empty slots`
- Слот relay-линии может быть занят устройством `Прочее оборудование`
- Источник relay-устройств: `wired_devices` с `connection_type = relay`
- Балансировщик relay-устройств переносит `220pump`/`pump-220v` с `connection_type = relay` в relay-линии
- В relay-линиях `pump-220v` подписывается как `Насос 220V N`, `boiler-pump` подписывается как `Насос бойлера N`
- Для relay-линий `rl6` и `rl6s` насосы в слотах `1`, `2`, `3` используют SVG с правым портом, а в слотах `4`, `5`, `6` используют SVG с левым портом:
  - `boiler-pump`: `boilerPumpRightPort.svg` / `boilerPumpLeftPort.svg`
  - `pump-220v`: `220pumpRightPort.svg` / `220pumpLeftPort.svg`
- Изображения `Прочего оборудования`:
  - по умолчанию: `resources/assets/engineerings/otherEquipment/otherEquipmentRightPort.svg`
  - альтернатива: `resources/assets/engineerings/otherEquipment/otherEquipmentLeftPort.svg`
  - выбор стороны порта выполняется автоматически по расположению слота относительно контроллера
- Коммутация `Прочего оборудования`:
  - для `otherEquipmentLeftPort.svg` линия выходит из порта сразу влево на `1 * indent`, затем выполняется вынос вниз
  - для `otherEquipmentRightPort.svg` линия выходит из порта сразу вправо на `1 * indent`, затем выполняется вынос вниз
- Для `smart2` и `pro`:
  - линия располагается под power-линией
  - правый край relay-слота лежит на вертикали правого края power-линии
  - если `ups` есть: relay-слот расположен на `7 * indent` ниже нижней точки `battery`
  - если `ups` нет: relay-слот расположен на `7 * indent` ниже нижней линии `power-unit`
- Для `go` и `go+`:
  - линия располагается справа от контроллера
  - нижняя линия relay-слота смещена на `3 * indent` ниже нижней линии контроллера
  - горизонтальный отступ от правой границы контроллера: `4 * indent`
- Список устройств relay-линии будет задан отдельно
- При включенном `Show line frames` relay-линия выделяется рамкой, как и другие линии
- Для `pro` слоты relay-линии располагаются справа налево (первый слот у правого края линии)
- У контроллера `pro` в собственной relay-линии `4` слота; балансировщики не должны размещать в `controller.relay_devices` больше четырёх relay-портов
- Для `pro` вынос вниз коммутационной relay-линии от нижней границы слота:
  - слот 1: `1 * indent`
  - слот 2: `2 * indent`
  - слот 3: `3 * indent`
  - слот 4: `4 * indent`
- Если сервопривод занимает порты `RELAY-2-B` и `RELAY-3-B` обычной `RELAY` линии, длинные горизонтальные участки обеих линий поднимаются на `3 * indent`
- У занятого relay-слота фон и обводка скрыты по умолчанию; отображаются только при hover
- У занятого relay-слота при hover отображается кнопка удаления (круг с крестиком)
- Занятые слоты RELAY-линии контроллера можно перемещать drag/drop; при перемещении линии коммутации перестраиваются к новой позиции слота
- Занятый relay-слот использует порт `RELAY-IN` для коммутации с портом `RELAY-1-B` контроллера
- При включенном `Show ports` порт `RELAY-IN` занятого relay-слота подсвечивается красной точкой

### Минимальные выносы для BUS-портов контроллера

- Правило действует для всех контроллеров, кроме `ecosmart`
- Для всех контроллеров, кроме `ecosmart`, позиция BUS-слота по `X` выравнивается относительно `BUS-A` порта контроллера:
  - правый край слота находится на `0.5 * indent` левее вертикали порта `BUS-A`
  - правило одинаково для пустого и занятого BUS-слота
- Пустой BUS-слот отображается только при включённом `Show empty slots`
- Для соединений BUS минимальный уровень горизонтального участка линии должен быть не выше:
  - Для `go`, `go+`, `smart2`:
    - `BUS-A`: `1 * indent` ниже нижней линии контроллера
    - `BUS-B`: `2 * indent` ниже нижней линии контроллера
  - Для `pro`:
    - `BUS-A`: `3 * indent` ниже нижней линии контроллера
    - `BUS-B`: `4 * indent` ниже нижней линии контроллера
- Если вынос, рассчитанный от целевого BUS-устройства (котла), меньше указанного минимума от контроллера, используется минимум от контроллера

## EXT линия (ext line)

- `EXT` — отдельный вид соединения в системе
- Линия доступна только для контроллеров `pro` и `ecosmart`
- Линия располагается справа от контроллера
- По вертикали линия находится в одном ряду с контроллером с выравниванием по нижнему краю

### Устройства EXT линии

- По `EXT` линии могут подключаться только устройства следующих типов:
  - `bl2`
  - `rl6`
  - `rl6s`
  - `io4`
  - `di6`

### Дополнительный отступ EXT для `pro`

- Если у контроллера `pro` линия `1-wire` не пустая, к расстоянию между контроллером и первым модулем `EXT` добавляется `8 * indent`
- Базовый отступ первого `EXT` модуля сохраняется (`4 * indent`), дополнительный применяется поверх него
- Дополнительный отступ влияет на базовую X-позицию всей `EXT` линии, поэтому сохраняются штатные расстояния между всеми соседними `EXT` модулями

### RELAY-линии модуля `rl6`

- У модуля `rl6` есть две собственные RELAY-линии:
  - первая линия расположена слева выше модуля
  - вторая линия расположена справа выше модуля
- Нижняя грань каждой RELAY-линии находится на уровне верхней грани модуля `rl6`
- В каждой линии по `3` слота
- Слоты в линии расположены вертикально с зазором `4 * indent`
- Размер пустого слота: `8 * indent` на `8 * indent`
- Размер занятого слота для `Прочего оборудования` и `Насос 220V`: `8 * indent` на `8 * indent`
- Размер занятого слота для `stupidBoiler`: `6 * indent` на `10 * indent`
- Занятые слоты имеют инфоблоки над собой
- Если включён `Show empty slots` или хотя бы один порт `RELAY` модуля занят, слева от `rl6` требуется дополнительный отступ `9 * indent`
- Если включён `Show empty slots` или хотя бы один порт `RELAY` модуля занят, справа от `rl6` требуется дополнительный отступ `20 * indent` только если на 1-wire линии этого же модуля нет устройств
- Из портов занятых слотов линии выходят без выноса вверх или вниз
- От порта устройства линия сразу идёт вправо до вертикали порта модуля для слотов `1`, `2`, `3`, затем спускается вниз в порт модуля
- От порта устройства линия сразу идёт влево до вертикали порта модуля для слотов `4`, `5`, `6`, затем спускается вниз в порт модуля
- Порты `RELAY-1-2-3-B` и `RELAY-4-5-6-B` в SVG разворачиваются в виртуальные порты `RELAY-1-B..RELAY-6-B` для коммутации
- Индикаторы `RELAY-INDICATOR-1..6` в SVG `rl6` отражают занятость соответствующих relay-слотов: занято — `#00DA00` с glow, свободно — `#D2D2D2`
- Сервопривод `220servo` с `connection_type = double_relay` занимает два соседних слота RELAY-линии модуля `rl6`

### RELAY-S-линии модуля `rl6s`

- У модуля `rl6s` есть две собственные RELAY-S-линии с расположением и размерами слотов как у RELAY-линий `rl6`
- Используются порты `RELAY-S-*`
- Порты `RELAY-S-1-2-3-B` и `RELAY-S-4-5-6-B` в SVG разворачиваются в виртуальные порты `RELAY-S-1-B..RELAY-S-6-B` для коммутации
- Индикаторы `RELAY-S-INDICATOR-1..6` в SVG `rl6s` отражают занятость соответствующих relay-s слотов: занято — `#00DA00` с glow, свободно — `#D2D2D2`
- Сервопривод `220servo` имеет приоритет на RELAY-S линиях: сначала `controller.relay_s_devices`, затем `rl6s.relay_s_devices`; при размещении он занимает два соседних слота RELAY-S-линии
- Для `220servo` и `valve` на relay-линиях модулей `rl6`/`rl6s`: если устройство левее портов модуля, провод выходит сразу вправо до X порта и затем идёт вертикально в порт; если устройство правее портов модуля, провод выходит сразу влево до X порта и затем идёт вертикально в порт

### Канальные линии модуля `io4`

- `010servo` из `wired_devices` с `connection_type = di` всегда распределяется только на модуль `io4`
- `010pump` из `wired_devices` с `connection_type = di` всегда распределяется только на модуль `io4`
- Целевая линия: `io4.channel_devices`
- Успешно распределённый `010servo`/`010pump` удаляется из верхнего `wired_devices`
- Если на `io4` нет свободного канала, объект остаётся в `wired_devices`
- Если у `010servo.additions` есть `mixing-ntc-sensor` с `connection_type = ntc`, балансировщик `010servo` переносит и сервопривод, и этот NTC-датчик на один и тот же модуль `io4`
- Если у `010pump.additions` есть `mixing-ntc-sensor` с `connection_type = ntc`, балансировщик переносит и насос, и этот NTC-датчик на один и тот же модуль `io4`
- `010servo`/`010pump` занимает один слот `io4.channel_devices`; вложенный NTC-датчик занимает следующий слот `io4.channel_devices` того же модуля
- Размещённый NTC-датчик удаляется из `additions` сервопривода в материализованной схеме, чтобы не было дублирования
- SVG для `010servo`: `resources/assets/servo/010servoLeftPorts.svg`
- Коммутация `010servo` с `io4` выполняется одним проводом к порту `CHANNEL-IN` в SVG сервопривода
- Логические слоты CHANNEL-линии `io4` расположены сверху вниз по каналам `1..4`
- Если `Show empty slots` выключен, занятые CHANNEL-слоты отрисовываются компактно ближе к модулю, сохраняя порядок каналов сверху вниз
- Если `Show empty slots` включён, все 4 CHANNEL-слота отображаются на штатных позициях по своим индексам

## Power линия (power line)

- `Power` — линия модулей питания
- Линия присутствует только у контроллеров `pro` и `smart2`
- Линия строится по данным `incomingScheme.power_modules`
- Базовый зазор между элементами power линии: `4 * indent`

### Обязательные модули питания

- Для `pro` и `smart2` модули `circuit-breaker` и `power-unit` отображаются всегда
- Эти модули считаются обязательными и должны присутствовать в `incomingScheme.power_modules`
- Порядок в линии: автоматический выключатель (`circuit-breaker`) всегда левее блока питания (`power-unit`)

### UPS в power линии

- Модуль `ups` может быть отображён из `incomingScheme.power_modules`
- Также `ups` может быть добавлен пользователем через пустой слот (`+`) в power линии

### Battery при наличии UPS

- Если `ups` присутствует в системе, модуль `battery` отображается всегда
- `battery` располагается прямо под `ups`
- Смещение `battery` по `Y`: на `10 * indent` ниже нижней линии `ups`
- Порты `battery` участвуют в коммутации

### Коммутация power линии

- `circuit-breaker <-> power-unit`:
  - `L-OUT` -> `L-IN`
  - Линия бордового цвета
  - Вынос вниз: `1 * indent` от нижнего края пары модулей

- `ups <-> controller` (если `ups` есть):
  - `12VDC-OUT-V+` <-> `12VDC-IN-V+` (красный)
  - `12VDC-OUT-GND` <-> `12VDC-IN-GND` (чёрный)
  - `DI-OUT-1` <-> `DI-IN-1` (синий)
  - `DI-OUT-2` <-> `DI-IN-2` (синий)

- `ups <-> power-unit` (если `ups` есть):
  - `12VDC-IN-GND` (ups) <-> `12VDC-OUT-GND` (power-unit)
  - `12VDC-IN-V+` (ups) <-> `12VDC-OUT-V+` (power-unit)
  - Вынос линий вверх

- `battery <-> ups`:
  - `ACID-BAT-V+` <-> `ACID-BAT-V+`
  - `ACID-BAT-GND` <-> `ACID-BAT-GND`
  - Правило трассировки от батареи:
    - `V+`: сначала вверх на `2 * indent` выше верхней линии `battery`, затем вправо к X порта `ups`, затем в порт `ups`
    - `GND`: сначала вверх на `1 * indent` выше верхней линии `battery`, затем вправо к X порта `ups`, затем в порт `ups`

- `power-unit <-> controller` (только если `ups` отсутствует):
  - `12VDC-OUT-V+` <-> `12VDC-IN-V+`
  - `12VDC-OUT-GND` <-> `12VDC-IN-GND`
  - Специфическая трассировка:
    - от `power-unit` линии идут вверх (`V+` на `2 * indent`, `GND` на `1 * indent`)
    - затем вправо от `power-unit` (`V+` на `2 * indent`, `GND` на `1 * indent`)
    - затем вниз до уровня под контроллером (`V+` на `1 * indent` ниже нижней линии контроллера, `GND` на `2 * indent`)
    - затем вправо под контроллером до X целевого порта и в порт контроллера

### Порт `N` у power-unit

- У `power-unit` порт класса `N` является входом питания и не коммутируется с другими устройствами
- Из порта `N` всегда отрисовывается вертикальная линия вниз
- Конец линии: на уровне `2 * indent` ниже нижней линии `power-unit`
- Сразу под концом линии отображается символ `N`

### Порт `L-IN` у circuit-breaker

- У `circuit-breaker` порт класса `L-IN` не коммутируется с другими устройствами
- Из порта `L-IN` всегда отрисовывается вертикальная линия вверх
- Конец линии: на уровне `2 * indent` выше верхней линии `circuit-breaker`
- Сразу над концом линии отображается символ `L`

## Понятие "порт"

**Порт** — это тег `<path>` (или другой) внутри SVG каждого элемента системы (контроллера, модуля, устройства), содержащий атрибут `class`. Порты обозначают точки соединения элемента с другими элементами системы.

- Каждый контроллер, модуль и устройство содержит внутри своего SVG-файла теги с классами — это порты соединения
- Порты используются для определения точек подключения линий связи между элементами
- Название порта = значение атрибута `class`

### Порты контроллеров (по результатам анализа SVG)

**go** (`go.svg`): `1-WIRE-V+`, `1-WIRE-DAT`, `1-WIRE-GND`, `BUS-A`, `BUS-B`, `RELAY-1-COM`, `RELAY-1-NO`

**smart2** (`smart2.svg`): `1-WIRE-V+`, `1-WIRE-DAT`, `1-WIRE-GND`, `12VDC-IN-V+`, `12VDC-IN-GND`, `BUS-A`, `BUS-B`

**pro** (`pro.svg`): `DI-IN-1`, `BUS-A`, `BUS-B`, `1-WIRE-V+`, `1-WIRE-DAT`, `1-WIRE-GND`, `EXT-OUT-A`, `RELAY-1-A`, `RELAY-1-B`, `RELAY-2-A`, `RELAY-2-B`, `RELAY-3-A`, `RELAY-3-B`, `RELAY-4-A`, `RELAY-4-B`

### RELAY-порты контроллера `pro` (подвод линии)

- Порты `RELAY-1-A`, `RELAY-2-A`, `RELAY-3-A`, `RELAY-4-A` не коммутируются с внешними устройствами
- Из каждого `RELAY-*-A` рисуется вертикальная линия вниз на `4 * indent` и подпись `L` под концом линии
- Линия и подпись для конкретного `RELAY-*-A` показываются только если соответствующий `RELAY-*-B` считается занятым

### RELAY-порт `A` у `smart2` (подвод линии)

- Для `smart2` порт `RELAY-1-A` также используется как подвод линии
- Из `RELAY-1-A` рисуется вертикальная линия вниз на `2 * indent` и подпись `L` под концом линии
- Линия и подпись отображаются только если соответствующий `RELAY-1-B` занят
- Для `pro` используются индикаторы `RELAY-INDICATOR-1..4`:
  - `RELAY-1-A/RELAY-1-B` <-> `RELAY-INDICATOR-1`
  - `RELAY-2-A/RELAY-2-B` <-> `RELAY-INDICATOR-2`
  - `RELAY-3-A/RELAY-3-B` <-> `RELAY-INDICATOR-3`
  - `RELAY-4-A/RELAY-4-B` <-> `RELAY-INDICATOR-4`
- Цвет индикатора зависит от занятости соответствующего relay-слота:
  - занято: `#00DA00` с glow `0 0 6px #00DA00`
  - не занято: `#D2D2D2`

## Глобальное правило — отсутствие наложений

Модули, устройства и информационные блоки **не должны накладываться друг на друга**. Каждый элемент располагается строго в своей области холста без пересечений с другими элементами.

- Правило обязательно для **всех** перемещаемых устройств и модулей: `wireless`, `1-wire`, `ext`
- При перетаскивании, если новая позиция даёт пересечение с любым другим устройством/модулем или с контроллером, позиция считается невалидной
- Невалидная позиция не сохраняется: элемент откатывается в последнюю валидную позицию

## Инварианты (нельзя нарушать)

- Элементы схемы, инфоблоки и контроллер не должны пересекаться
- Невалидный drag не сохраняется, элемент откатывается в последнюю валидную позицию

- `Show empty slots` показывает только пустые слоты (и one-wire слот у термостата)
- Обязательный порядок в power линии: `circuit-breaker` левее `power-unit`
- При наличии `ups` модуль `battery` обязателен и отображается всегда
- Все балансировщики должны переносить объекты из исходных верхнеуровневых списков в целевые линии, а не дублировать их; успешно размещённый объект удаляется из исходного списка
- Если объект не удалось разместить из-за отсутствия ёмкости линии, он должен остаться в исходном списке
- Добавление устройства через пустой слот выполняется непосредственно в ту целевую линию, в которую это устройство распределил бы балансировщик; добавление из слота не должно создавать дубликат в верхнеуровневом исходном списке
- Специальные `ecosmart` controller-линии (`220_servo_devices`, `relay_s_valve_devices`, `relay_boiler_gvs_devices`, `relay_220pump_devices`, `relay_220pump5_devices`, `relay_220pump3_devices`, `leak_sensor_devices`, `strategy_sensor_devices`, `boiler_sensor_devices`, `mixing_ntc_devices`, `ecosmart_bl2`) являются внутренними render-полями и не должны попадать в публичный `incomingScheme`/save; при сериализации устройства возвращаются в `wired_devices` или `sensors`

## Контракт входных данных (актуальный)

```js
const incomingScheme = {
    controller: 'go | go+ | smart2 | ecosmart | pro',
    wireless_devices: [
        // { id?, type, color?, additions? }
    ],
    wired_devices: [
        // используется минимум для type=thermostat в 1-wire линии
    ],
    ext_modules: [],
    di_modules: [],
    one_wire_modules: [], // поддерживается alias rdt -> rdt2
    power_modules: ['circuit-breaker', 'power-unit'],
    wifi_modules: []
};
```

### Нормализация и alias

- Для 1-wire модулей поддерживается alias: `rdt` -> `rdt2`
- После первичной балансировки контроллерная 1-wire линия хранится в `controller.one_wire_devices`, EXT-линии — в `ext_modules[n].one_wire_devices`, а `one_wire_modules` очищается
- `controller_one_wire_devices` больше не является актуальным хранилищем; оно допускается только как legacy fallback входных данных при отсутствии `controller.one_wire_devices`
- `device_type` — новый мета-параметр объектов для будущего автоподбора контроллера и модулей; текущая отрисовка его не использует
- Для power модулей учитывается нормализация:
  - `circuitbreaker` -> `circuit-breaker`
  - `powerunit` -> `power-unit`
- Для EXT модулей актуальная форма входных данных — объект `{ id?, type, connection_type: 'EXT' }`; строковая форма поддерживается только как legacy fallback
- Для беспроводного термостата неизвестный/пустой `color` -> `black`

## Матрица коммутаций (кратко)

### 1-wire

- `1-WIRE-V+` -> `1-WIRE-V+` (красный), вынос `1 * indent` от целевого слота
- `1-WIRE-DAT` -> `1-WIRE-DAT` (жёлтый), вынос `2 * indent`
- `1-WIRE-GND` -> `1-WIRE-GND` (чёрный), вынос `3 * indent`
- Исключение: если целевой слот — термостат, линия приходит на уровень порта термостата

### Thermostat <-> Floor sensor

- `1-WIRE-GND`: вниз `3 * indent`, затем по X, затем в порт
- `1-WIRE-DAT`: вниз `2 * indent`, затем по X, затем в порт
- `1-WIRE-V+`: вниз `1 * indent`, затем по X, затем в порт

### Power

- `circuit-breaker:L-OUT` -> `power-unit:L-IN` (бордовый), вынос вниз `1 * indent`
- Если `ups` есть:
  - `ups:12VDC-OUT-V+` -> `controller:12VDC-IN-V+` (красный)
  - `ups:12VDC-OUT-GND` -> `controller:12VDC-IN-GND` (чёрный)
  - `ups:DI-OUT-1` -> `controller:DI-IN-1` (синий)
  - `ups:DI-OUT-2` -> `controller:DI-IN-2` (синий)
  - `power-unit:12VDC-OUT-*` -> `ups:12VDC-IN-*` (вынос вверх)
- Если `ups` нет:
  - `power-unit:12VDC-OUT-V+` -> `controller:12VDC-IN-V+`
  - `power-unit:12VDC-OUT-GND` -> `controller:12VDC-IN-GND`
  - Используется специальная многошаговая трассировка (см. раздел Power выше)
