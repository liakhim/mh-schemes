# Connection Layout Rules

`connection_layout` сохраняет физический порт между save/reopen, не превращая внутренние линии в публичный источник истины.

## Contract

```js
connection_layout: {
    version: 1,
    assignments: [{
        device_type: 'discrete_signal',
        device_id: 'fire-input',
        owner_kind: 'ext_module',
        owner_id: 'di6-main',
        line: 'channel_devices',
        slot: 4,
    }],
}
```

- Устройство продолжает храниться в `boilers`, `wired_devices` или `sensors`.
- `device_type` и `device_id` однозначно находят устройство. Назначение записывается только при наличии реального `id`.
- Тип значения `device_id` является частью identity: числовой `1` и строковый `'1'` обозначают разные устройства. Полностью одинаковые `type` и `id` по-прежнему считаются одной публичной копией.
- `owner_kind` принимает `controller`, `ext_module`, `di_module`.
- `owner_id` обязателен для модулей и не используется для controller. Позиция и индекс модуля в массиве никогда не являются идентификатором владельца.
- `line` - имя материализованной линии; `slot` - физический индекс с нуля.
- Для `double_relay` `slot` обозначает первый порт непрерывной пары и восстанавливается как `relay_slot_index`.
- Одно устройство получает не более одного assignment.
- Группа IO4 из `010servo`/`010pump` и его `mixing-ntc-sensor` additions сохраняет отдельный assignment на каждый физический канал. Parent и все child assignments должны указывать один IO4 и последовательные слоты, начиная со слота parent.

## Load And Fallback

Восстановление выполняется в `materializeBalancedOneWireScheme` до всех соответствующих балансировщиков. Assignment применяется только после проверки версии layout, существования устройства в публичном bucket, владельца по ID, типа модуля, допустимой линии, диапазона и занятости слота, а также совместимости устройства с линией.

После успешного размещения удаляется ровно найденная копия устройства из одного публичного bucket. Поэтому последующий балансировщик не создаёт дубль.

Невалидный assignment не меняет публичные данные. Повреждённый слот, несовместимая линия, изменённый тип или удалённый owner module приводят к обычной автоматической балансировке. Старые схемы без `connection_layout` также полностью используют прежний pipeline.

IO4-группа восстанавливается атомарно. Если отсутствует или повреждён хотя бы один child assignment, parent не закрепляется частично: устройство остаётся в `wired_devices` вместе с публичными `additions`, и `balance010Servos` заново размещает всю contiguous-группу без потерь и дублей.

## Save

`serializePublicScheme` сначала собирает назначения из фактических controller, EXT и DI module lines, затем сворачивает устройства обратно в публичные buckets. Старое значение `connection_layout` не копируется. При отсутствии назначений поле удаляется.

Version 1 покрывает controller BUS/RELAY/RELAY-S/DI/4-20, EXT BL2/RL6/RL6S/IO4/DI6 и relay-линии DI-модулей RL2/RL2S. EXT BL2 допустим только для `pro`: assignment обычного BL2 у `ecosmart` не сохраняется и не восстанавливается. Специальные ECOsmart линии, включая `controller.ecosmart_bl2`, и 1-wire ownership намеренно остаются в существующих специализированных pipeline.
